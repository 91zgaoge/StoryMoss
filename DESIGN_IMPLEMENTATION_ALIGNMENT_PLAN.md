# StoryForge (草苔) v5.6.4 设计-实现对齐全面分析与实施计划

> **项目概述**: StoryForge 是一个基于 Tauri + Rust + React 的 AI 导演式小说创作系统，采用创新的"幕前-幕后"双界面架构，专为小说作者打造导演式创作工作台。

## 📋 执行摘要

经过全面检视项目设计文档、架构规范和当前实现代码，识别出 **12 项关键设计-实现差距**，主要集中在三个核心维度：

1. **幕前幕后自动关联** (5项差距) - 数据变更未能在双界面间实时同步
2. **后台自动化闭环** (5项差距) - 智能化创作飞轮缺乏持续反馈机制  
3. **系统整洁度与可观测性** (2项差距) - 部分功能存在死胡同和不一致性

**影响评估**: 这些差距导致用户体验割裂、AI 学习效果递减、数据一致性问题，严重影响了"越写越懂"的核心价值主张。

**修复策略**: 采用增量式修复方案，对现有 217 个 Rust 单测和前端测试零破坏，通过 5 个步骤系统性解决所有差距。

## 🎯 项目设计目标分析

### 核心设计理念

StoryForge 的设计围绕三个核心理念：

#### 1. 剧院式双界面架构
- **幕前 (Frontstage)**: 沉浸式写作界面，如同登台演出
  - 极简阅读写作界面，暖色调 (#f5f4ed) Claude 阅读体验
  - AI 萤火随行，三态文思模式 (off/passive/active)
  - 内联 `/` 指令，幽灵文本续写
  
- **幕后 (Backstage)**: 专业工作室，如同后台准备
  - 故事/场景/角色管理，知识图谱可视化
  - 技能系统，MCP 外部服务器，数据导出
  - 创作方法论引擎，StyleDNA 系统

#### 2. 场景化叙事系统
- 以场景为单位的戏剧冲突驱动
- Scene 分 stage 编辑（规划/大纲/起草/审校/定稿）
- 伏笔看板全生命周期管理

#### 3. 增强记忆系统 ("越写越懂")
- Layer 4: Multi-Agent Sessions (世界观/人物/文风助手)
- Layer 3: Knowledge Graph (带权实体关系图谱)  
- Layer 2: Vector Store (CJK分词语义检索)
- Layer 1: Raw Sources (场景正文、角色设定)

### 关键设计目标

根据架构文档和版本更新日志，项目的关键设计目标包括：

1. **幕前幕后无缝协作**: 双窗口实时数据同步，任何变更零延迟对齐
2. **后台自动化闭环**: IngestPipeline、向量索引、知识图谱、伏笔追踪形成自维持反馈环
3. **智能化创作飞轮**: 自适应学习系统记录反馈、挖掘偏好、动态调节生成参数
4. **创世引擎**: 输入一句话自动生成完整小说世界（大纲、角色、场景、伏笔）
5. **StyleDNA 系统**: 六维定量风格模型，实时风格匹配
6. **工作流引擎**: 7 阶段全自动创作工作流

## 🚨 识别的设计-实现差距

基于对 `.kiro/specs/design-implementation-alignment-v5.7/` 目录下文档的分析，识别出以下 12 项关键差距：

### P0 级别差距 (数据层正确性)

#### 1.1 SQLite 外键约束失效
**问题**: 连接池未开启 `PRAGMA foreign_keys`，声明式 `ON DELETE CASCADE` 被静默忽略
**影响**: 删除故事后关联数据残留，造成幽灵数据
**位置**: `src-tauri/src/db/connection.rs::init_db`

#### 1.2 故事删除级联不完整  
**问题**: `delete_story` 不在事务内显式清理关联表
**影响**: chapters/scenes/kg_entities 等表中 story_id 残留
**位置**: `src-tauri/src/db/repositories_v3.rs`

#### 1.3 角色删除关联清理缺失
**问题**: `delete_character` 未清理 character_relationships 表
**影响**: 关系页面显示"未知角色"，控制台警告
**位置**: `src-tauri/src/db/repositories_v3.rs`

#### 1.4 角色关系 CRUD IPC 缺失
**问题**: 缺少 `create_character_relationship` 等 IPC 命令
**影响**: 前端无法操作角色关系，功能不可用
**位置**: `src-tauri/src/lib.rs` 或 `commands_v3.rs`

#### 1.5 知识图谱变更同步事件缺失
**问题**: KG 实体/关系变更不触发 `sync-event`
**影响**: KG 可视化页面不实时刷新
**位置**: `src-tauri/src/state_sync/mod.rs`

### P1 级别差距 (后台自动化闭环)

#### 1.6 Ingest 完成不触发 KG 刷新
**问题**: `auto_ingest_chapter` 成功后不发射 KG 同步事件
**影响**: 新抽取的实体不在 KG 图中显示
**位置**: `src-tauri/src/memory/ingest.rs`

#### 1.7 计划模板库仅内存存储
**问题**: `PlanTemplateLibrary` 重启后丢失学习成果
**影响**: "越写越懂"效果递减，重复 LLM 调用
**位置**: `src-tauri/src/planner/template_learning.rs`

#### 1.8 能力进化仅启动时触发
**问题**: 能力进化只在启动时执行一次，缺少周期触发
**影响**: 长时间使用后 AI 能力描述不更新
**位置**: `src-tauri/src/capabilities/evolution.rs`

#### 1.9 伏笔账本缓存同步缺失
**问题**: payoff_ledger 更新不触发前端缓存失效
**影响**: 伏笔状态变更不实时反映在 UI
**位置**: `src-frontend/src/hooks/useSyncStore.ts`

#### 1.10 工作流实例重启恢复不完整
**问题**: 重启后 Pending/Running 实例不重新入队
**影响**: 中断的工作流需手动重启
**位置**: `src-tauri/src/workflow/mod.rs`

### P2 级别差距 (可观测性与整洁度)

#### 1.11 图像模型配置死胡同
**问题**: `image` 类型 LLM Profile 可创建但不可用
**影响**: 用户困惑，配置无效占位
**位置**: 前端 LLM Profile 创建 UI

#### 1.12 KG 更新路径不统一
**问题**: 部分 KG 更新不经过 `StateSync`
**影响**: 同步行为不一致，难以调试
**位置**: 多个 KG 操作入口

## 🛠️ 修复实施计划

### 修复策略

采用**增量式修复**方案，遵循以下原则：
- 对现有 217 个 Rust 单测零破坏
- 保持 v5.6.4 已验证行为不变
- 优先修复 P0 级别数据正确性问题
- 分步骤验证，每步都可独立测试

### 实施步骤

#### Step A: 数据层根因修复 (P0)
**目标**: 修复 SQLite 外键约束和级联删除问题

**具体任务**:
1. 修改 `SqliteConnectionManager::with_init` 开启 `PRAGMA foreign_keys = ON`
2. 加固 `delete_story` 事务清理逻辑
3. 加固 `delete_character` 关联清理逻辑
4. 添加数据一致性测试用例

**预期结果**: 删除操作正确级联，无幽灵数据残留

#### Step B: 同步事件扩展 (P0)
**目标**: 补齐缺失的同步事件

**具体任务**:
1. `StateSync` 新增 4 个 emit 方法:
   - `emit_knowledge_graph_updated`
   - `emit_character_relationships_updated` 
   - `emit_payoff_ledger_updated`
   - `emit_ingestion_completed`
2. `useSyncStore` 新增对应 case 处理
3. 相关操作点调用新增的 emit 方法

**预期结果**: 所有数据变更都能触发前端缓存刷新

#### Step C: CRUD 缺口补齐 (P0)
**目标**: 补齐角色关系 CRUD IPC 命令

**具体任务**:
1. 实现 `create_character_relationship` IPC
2. 实现 `update_character_relationship` IPC  
3. 实现 `delete_character_relationship` IPC
4. 前端调用接口适配

**预期结果**: 角色关系功能完全可用

#### Step D: 后台自动化闭环 (P1)
**目标**: 建立持续的自动化反馈机制

**具体任务**:
1. `PlanTemplateLibrary` 持久化到新表 `plan_templates` (Migration 43)
2. 能力进化改为周期 + 阈值触发机制
3. `WorkflowEngine::with_pool` 启动后重新入队恢复实例
4. Ingest 完成触发 KG 刷新事件

**预期结果**: 后台自动化形成自维持反馈环

#### Step E: 系统整洁度优化 (P2)
**目标**: 消除死胡同和不一致性

**具体任务**:
1. `image` Profile UI 添加"暂未实现"标注或阻止创建
2. 统一所有 KG 更新路径经过 `StateSync`
3. 添加相关文档和注释

**预期结果**: 系统行为一致，用户体验清晰

### 验证计划

#### 功能验证
1. **数据一致性验证**: 删除故事/角色后检查关联表无残留
2. **同步事件验证**: 各类数据变更后检查前端缓存自动刷新
3. **CRUD 完整性验证**: 角色关系的增删改查全流程测试
4. **自动化闭环验证**: 重启后检查学习成果保持、工作流恢复

2. **内存使用**: 监控新增缓存和队列的内存占用
3. **响应时间**: 确保同步事件不影响 UI 响应性

#### 兼容性验证
1. **向后兼容**: 现有数据库和配置文件正常迁移
2. **API 兼容**: 前端调用接口保持不变
3. **测试兼容**: 现有 217 个单测全部通过

## ⚠️ 风险评估与缓解

### 技术风险
**风险**: 数据库迁移可能影响现有数据
**缓解**: 
- 新增表结构，不修改现有表
- 完整的备份和回滚机制
- 分步骤验证，每步可独立回退

**风险**: 同步事件频繁触发可能影响性能  
**缓解**:
- 事件去重和批量处理
- 异步发射，不阻塞主流程
- 性能监控和阈值告警

### 业务风险
**风险**: 修复过程中可能引入新 bug
**缓解**:
- 增量式修复，每步独立验证
- 完整的测试覆盖
- 灰度发布和快速回滚

**风险**: 用户学习成本增加
**缓解**:
- 保持现有 UI 和交互不变
- 新功能渐进式引导
- 详细的更新说明文档
